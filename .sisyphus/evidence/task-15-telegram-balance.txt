DESIGN DECISIONS
================

1. In-memory UserRegistry for Phase 1
   - Rationale: MVP simplicity, no database dependency
   - TODO for production: PostgreSQL with user_id → account_id mapping

2. Mock responses instead of API calls
   - Rationale: Webhook handler only, no outbound calls per requirements
   - Balance: hardcoded SGD 1,234.56
   - Payment: mock 5-stage pipeline with realistic timings
   - TODO for production: Call AOA API gateway endpoints

3. IntentParser regex patterns
   - Rationale: Simple NLP for MVP, no ML/LLM dependency
   - Patterns: "pay 500 SGD to merchant-1", "send 1000 to agent-x"
   - Fallback: UNKNOWN intent with help text

4. PipelineDisplay symbols
   - Rationale: OMO-style visual feedback for Telegram (text-only)
   - Symbols: ✅ ⏳ ⬜ ❌ (Unicode, Telegram-compatible)

5. Hono framework
   - Rationale: Lightweight, Bun-native, matches api-gateway pattern
   - Route: POST /api/v1/telegram/webhook (consistent with API versioning)
